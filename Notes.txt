****Understanding Java Code****
Hadoop always works in 3 parts:-
(i)Mapper
Takes input → processes it → gives output (key, value)
(ii)Reducer
Takes mapper output → combines it → final result
(iii)Driver(Main)
Connects everything (Mapper + Reducer + Input + Output)


Firstly lets understand about the Mapper Class

public static class TokenizerMapper
    extends Mapper<Object, Text, Text, IntWritable> // <Input Key, Input Value, Output Key, Output Value>

In this we tell what type of data will come IN and what type of data will go Out


InputKey -> It represents the line number or byte offset(Use OBJECT in InputKey everytime)
InputValue -> Actual data from the file (Text)

OutputKey -> If output key will be in NUMERIC FORM use IntWritable else Text
OutputValue -> Same for this

Reducer Class

public static class Reducer
    extends Reducer<Text, IntWritable, Text, IntWritable> // <Input Key, Input Value, Output Key, Output Value>

Last two parameters of the Mapper class will be same for the first two parameters of the Reducer class



****Introduction to Hive****

->Understanding the problem
Before Hive existed, Hadoop users had a major problem. If they want to extract the data from the hadoop, they have to write the code with the help of Java MapReduce, create Mapper class, Reducer Class.
For example:-
Find total salary department-wise?
For this simple task they have to write a lot of Java code.

So to solve this problem "HIVE" was created.

->Definition
It is Data warehouse software built on top of Hadoop used for storing, managing, and analyzing big data using SQL-like queries.

->In Simple terms:-
HDFS = Storage
MapReduce = Processing Engine
Hive = SQL Layer over Hadoop

->Internal flow of HIVE
When we write queries:
    Select * from employee;
Hive internally converts it into:
    MapReduce job
        OR
    Tez job
        OR
    Spark job
Then Hadoop executes it.

->Does HIVE store data??
No.
The real data is stored inside HDFS.
It manages:-
(i)Metadata (Data about data)
(ii)Provide SQL like interface
(iii)Converts Queries into Processing Jobs

->What makes HIVE popular?
(i)SQL-like language (HiveQL)
(ii)Schema on Read
In SQL databases, first we create the table structure and then store the data, so schema is applied during writing (Schema on Write).
But in Hive, data is already stored in HDFS. When we create a Hive table and read the data, Hive applies the schema at that time. Therefore Hive follows Schema on Read.
(iii)Scalability
Hive handles huge datasets
(iv)Distributed processing

->Limitations
(i)Hive is not used for Real-Time systems (High Latency, Processing nature)
Because when we run a query using Hive, internally it does many things such as:-
- reads huge data from HDFS
- converts query into Hadoop jobs
- sends work to multiple machines
- processes large files
- combines results

->Important Points
(i)Hive is mainly designed for OLAP (Online Analytical Processing):-
Meaning:
- analyzing huge data
- generating reports
- business intelligence

Not:
OLTP (Online transaction Processing)

(ii)Hive is not a database like MySQL
It is a tool which hadoop uses to extract the information like SQL queries. (SQL layer over Hadoop)

****Bucketing****
It is a technique in HIVE where a table data is divided into fixed number of files called buckets.

It works using Hashing.

For exampke:-
Suppose table has:
id	name
1	Aman
2	Riya
3	Karan
4	Simran

Now suppose:- INTO 4 BUCKETS;

Hive will distribute rows into:
- bucket 1
- bucket 2
- bucket 3
- bucket 4
using:
hash function

Why Bucketing:-
For faster query performance, Efficient joins, Better organization

Now there is also one more thing which is "Partitioning":
- Bucketing divide the data using a HASH function.
- But Partitioning divides the data using column values(country = "India', country = "USA")

-> How Hashing Works??
Hive internally divides the rows using this formula
hash(id) % number of buckets

-> Example for Partitioning and Bucketing

emp_id	name	department	country
1	    Aman	IT	        India
2	    Rahul	HR	        India
3	    John	IT	        USA
4	    Alex	Sales	    USA
5	    Simran	HR	        India
6	    Bob	    IT	        USA

(i)Partitioning
Suppose we do Partitioning on country column.
Hive Physically Creates Separate Folders:-
    employee_data/
        country=India/
        country=USA/

Data Distribution
country=India
emp_id	name	department
1	    Aman	IT
2	    Rahul	HR
5	    Simran	HR

country=USA
emp_id	name	department
3	    John	IT
4	    Alex	Sales
6	    Bob	    IT

Benefits:- Faster performance (If you want to search anything realted to "India", it will search only the country="India" partition -> Query Becomes faster)

Partitioning creates folders

(ii)Bucketing
Now suppose inside each partition we create:
4 buckets
using:
    emp_id

Hive applies a hash function on emp_id column:- emp_id % (number of buckets)\

Data Distribution
Bucket 0
Rows where:
emp_id % 4 = 0

emp_id	name
4	    Alex

Bucket 1
emp_id	name
1	    Aman
5	    Simran

Bucket 2
emp_id	name
2	    Rahul
6	    Bob

Bucket 3
emp_id	name
3	    John

Bucketing creates files.

****Introduction to HBase****

Why Hbase??
Hadoop + HDFS are excellent for:-
- Storing HUGE data
- Batch processing
But they have one limitation:-
    Slow random read/ write access

For example:-
You want to fetch only one record instantly. Hadoop MapReduce will takes a lot of time and it is not ideal for this.
Because Hadoop is batch-processing oriented.

Why HBase Was Created??
HBase was created to provide:
real-time read/write access on Hadoop
This is the biggest reason.

- Definition:-
Apache HBase is a distributed, scalable, NoSQL database built on top of Hadoop HDFS.

- Simple way:-
HDFS -> Storage
Hive -> SQL analytics
HBase -> Real-time NoSQL database

Hive is best for:-
Analytics
Reports
Batch processing
SQL-like
OLAP

HBase is best for:-
Fast reads/writes
Real-time access
Random access
NoSQL
OLTP-like access


-> Column - oriented
Traditional databases store:
    rows together
But Hbase stores:
    columns together

For example:
Traditional DB:
id	name	age
1	Aman	21
Stored row-wise

HBase stores by columns:
    id column separately
    name column separately
    age column separately

Why it is better? (Good Performance -> Stores data by columns)
Suppose we only want "age" of a customer, we will go to only the "age" column. Traditional databases reads the entire rows, even though age is only needed.

-> HBase Architecture:-
HBase follows:
    Master-Slave Architecture

Meaning:
Component	        Role
HMaster	        Controller
RegionServer	Worker
HDFS	        Storage

Client
   ↓
HMaster
   ↓
RegionServer
   ↓
HDFS

(i)HMaster(Manager/ Controller):
- Table Management(Create, Delete, Schema Changes)
- Region Assignment (Which RegionServer handles which region)
- Monitoring (Checks server alive or not)

(ii)RegionServer (Stores and manages actual data)
Internal Components of RegionServer
- WAL (Write Ahead Log)
Before actual write:
    Data is first stored in WAL
Why:- If system crashes (Data can be restored)

- MemStore (in-memory write buffer)
- HFiles
    Finally data stored permanently as:
        HFiles in HDFS

(iii)Region (Part of a table) (In HBase tables get automatically splitted)
- Table split into regions
- Each region contains row ranges
- Automatically split

For Example

Suppose table has millions of rows.

HBase splits table into:

Region 1 -> rows A-F
Region 2 -> rows G-M
Region 3 -> rows N-Z

Why Regions Needed?
Because huge table:
    cannot efficiently stay in one place.
Regions allow:
    distribution
    scalability

- Automatic Splitting
As data grows:
    regions automatically split
This improves:
    scalability
    load balancing

(iv)ZooKeeper (Coordinator/ Watcher)
- coordination service
- tracks active servers
- failures
- leader election

- Leader Election
Suppose:
    current HMaster fails
ZooKeeper helps select:
    new active master

(v)HDFS (storage layer)
- stores actual HFiles
- durability
- replication


-> HBase Data Model
Traditional databases stores data like:
id	name	age
1	Aman	21
Fixed structure.

HBase Structure is Different
HBase data model is based on:
(Table, RowKey, Column Family, Column, Timestamp) → Value

(i)Table
Same idea as relational DB.

Example:
users
students
employees

A table contains:
    rows
    column families

(ii)Row Key
- unique identifier
- rows are stored in lexicographical sorting (Dictionary based sorting)
HBase accesses data mainly using:
    RowKey
This is the heart of HBase.

Example
Suppose:
RowKey	    name	    age
user1001	Aman	    21

(iii)Column Family
Column Family groups related columns together.
For example:-
RowKey	personal:name	personal:age	academic:marks	academic:grade
101	        Aman	        21	            89	            A
102	        Rahul	        22	            75	            B

Column Family 1: personal which contains "name" and "age"
Column Family 2: academic which contains "marks" and "grade"

(iv)Column Qualifier
Specific column name inside Family
name, age, marks and grade

(v)Cell (Single Value)
Intersection of row and column
Each cell also stores TIMESTAMP

-> HBase Filters
Filters are used to retrieve only required rows instead of scanning unnecesaary data

(i)Prefix Filter (Works on Row Key)
scan 'student', {
 FILTER => "PrefixFilter('user_')"
}

Return only rows starting with user_ RowKey

(ii)SingleColumnValue Filter (Works on Column Values)

scan 'student', {
 FILTER => "SingleColumnValueFilter(
   'info',
   'age',
   =,
   'binary:20'
 )"
}

Fetch rows where:
Part	        Meaning
info	    column family
age	        qualifier
=	        operator
binary:20	value

Output -> Only rows where age = 20

Why binary:20 ?
HBase internally stores values in:
    byte/binary format
So comparison uses:
    binary:value

(iii)Combining Filters (FilterList)
scan 'student', {
 FILTER => "FilterList(
   AND,
   PrefixFilter('user_'),
   SingleColumnValueFilter(
     'info',
     'age',
     =,
     'binary:20'
   )
 )"
}

-> TTL (Time-To-Live)
How long data should remain in HBase
TTL is applied on Column Family level

TTL => 3600
Data survives for 3600 seconds

- Create Table with TTL
create 'student', {NAME => 'info', TTL => 3600}
    NAME is just a keyword

TTL is applied on entire 'info' column family

create 'student', {NAME => 'info', TTL => 3600}, 'contact'
